Create `src/pam/ingestion/stores/entity_relationship_store.py` with the following structure:

**Data classes:**
- `EntityVDBRecord` dataclass: `name: str`, `entity_type: str`, `description: str`, `source_id: str`, `file_path: str | None = None`
- `RelationshipVDBRecord` dataclass: `src_entity: str`, `tgt_entity: str`, `rel_type: str`, `keywords: str`, `description: str`, `source_id: str`, `weight: float = 1.0`

**Index mapping functions:**
- `get_entity_index_mapping(embedding_dims: int) -> dict`: Maps `name` (keyword, used as ES doc ID), `entity_type` (keyword for filtering), `description` (text with standard analyzer for BM25 fallback), `embedding` (dense_vector, cosine, indexed), `content_hash` (keyword for skip-re-embed), `source_ids` (keyword array), `file_paths` (keyword array). Settings: 1 shard, 0 replicas (same as `pam_segments`).
- `get_relationship_index_mapping(embedding_dims: int) -> dict`: Maps `src_entity` (keyword), `tgt_entity` (keyword), `rel_type` (keyword), `keywords` (text, standard analyzer), `description` (text, standard analyzer), `embedding` (dense_vector, cosine, indexed), `content_hash` (keyword), `weight` (float), `source_ids` (keyword array). Settings: 1 shard, 0 replicas.

**Helper function:**
- `make_relationship_doc_id(src: str, rel_type: str, tgt: str) -> str`: Alphabetically sort src/tgt, return `"{sorted[0]}::{rel_type}::{sorted[1]}"`. This ensures A::REL::B == B::REL::A for undirected relationships while allowing multiple relationship types between the same entity pair (per user decision).

**EntityRelationshipVDBStore class:**
- `__init__(self, client: AsyncElasticsearch, entity_index: str, relationship_index: str, embedding_dims: int)`: Store client, index names, dims.
- `async ensure_indices(self) -> None`: For each index, check `client.indices.exists()`, create with mapping if not exists. Log creation via structlog.
- `async _filter_unchanged(self, index_name: str, doc_ids: list[str], new_hashes: list[str]) -> tuple[list[str], list[int]]`: For each doc_id, do an `mget` call to fetch existing `content_hash` fields. Compare against new_hashes. Return (texts_to_embed, indices_to_embed) for only those that changed. This avoids re-embedding unchanged descriptions (per user decision: hash embedding text content, compare against existing ES record hash, skip API call if unchanged). Use SHA-256 consistent with existing content hashing.
- `async upsert_entities(self, entities: list[EntityVDBRecord], embedder: BaseEmbedder, source_id: str) -> int`: If empty, return 0. Build embedding texts as `f"{e.name}\n{e.description}"` (LightRAG format per user decision). Compute SHA-256 hashes. Call `_filter_unchanged` to identify changed entities. Embed only changed texts via `embedder.embed_texts()`. Build bulk actions using `{"index": {"_index": entity_index, "_id": entity.name}}` with all fields including `source_ids: [source_id]` and the embedding + content_hash. Execute `client.bulk(operations=..., refresh="wait_for")`. Log count via structlog. Return upserted count.
- `async upsert_relationships(self, relationships: list[RelationshipVDBRecord], embedder: BaseEmbedder, source_id: str) -> int`: Same pattern as entities but: embedding text = `f"{r.keywords}\t{r.src_entity}\n{r.tgt_entity}\n{r.description}"` (LightRAG format per user decision). ES doc ID via `make_relationship_doc_id(r.src_entity, r.rel_type, r.tgt_entity)`. Include `weight` field. Return upserted count.

**Config changes in `src/pam/common/config.py`:**

Use structlog for all logging. Import `BaseEmbedder` from `pam.ingestion.embedders.base`. Use `hashlib.sha256` for content hashing (consistent with project pattern).

**After the chunk loop, before Phase 3**, add VDB upsert step (only if both `vdb_store` and `embedder` are provided):
```python
if vdb_store is not None and embedder is not None:
    from pam.ingestion.stores.entity_relationship_store import (
        EntityVDBRecord,
        RelationshipVDBRecord,
    )

    # Build entity records from new_entities dict
    entity_records = [
        EntityVDBRecord(
            name=name,
            entity_type=info.get("type", "Unknown"),
            description=info.get("summary", ""),
            source_id=source_id,
        )
        for name, info in new_entities.items()
    ]

    # Build relationship records from accumulated edges
    rel_records = [
        RelationshipVDBRecord(
            src_entity=e["src_entity"],
            tgt_entity=e["tgt_entity"],
            rel_type=e["rel_type"],
            keywords=e["rel_type"].replace("_", " ").lower(),
            description=e["description"],
            source_id=source_id,
            weight=float(len(e["episodes"])) if e["episodes"] else 1.0,
        )
        for e in all_edges.values()
    ]

    try:
        entities_upserted = await vdb_store.upsert_entities(entity_records, embedder, source_id)
        rels_upserted = await vdb_store.upsert_relationships(rel_records, embedder, source_id)
        logger.info(
            "vdb_upsert_complete",
            doc_id=str(doc_id),
            entities_upserted=entities_upserted,
            relationships_upserted=rels_upserted,
        )
    except Exception:
        # VDB upsert is non-blocking, same as graph extraction itself
        logger.warning("vdb_upsert_failed", doc_id=str(doc_id), exc_info=True)
```

The VDB upsert is wrapped in try/except so failures don't affect the graph extraction result (consistent with the project's non-blocking graph extraction pattern from Phase 7).
