<objective>
Create the token-budgeted context assembly module that implements LightRAG's 4-stage pipeline pattern: (1) collect and normalize raw results, (2) per-category token truncation sorted by relevance, (3) chunk deduplication, (4) structured Markdown prompt construction.

Purpose: Transforms raw retrieval results into optimally organized, token-bounded context blocks so the LLM agent receives structured information (entities, relationships, source chunks) within predictable token limits instead of unbounded concatenated text.

Output: `src/pam/agent/context_assembly.py` module, tiktoken dependency, config settings, and unit tests.

**3. Create `src/pam/agent/context_assembly.py`:**

This module exports: `assemble_context`, `count_tokens`, `ContextBudget`, `AssembledContext`.

**ContextBudget dataclass:**
- `entity_tokens: int = 4000`
- `relationship_tokens: int = 6000`
- `max_total_tokens: int = 12000`
- `max_item_tokens: int = 500` (per-item description truncation cap -- Claude's discretion per CONTEXT.md)

**AssembledContext dataclass:**
- `text: str` (the final Markdown string)
- `entity_tokens_used: int`
- `relationship_tokens_used: int`
- `chunk_tokens_used: int`
- `total_tokens: int`

**Token counting (lazy singleton pattern):**
- Module-level `_encoder: tiktoken.Encoding | None = None`
- `_get_encoder()` function that lazily initializes with `tiktoken.get_encoding("cl100k_base")`
- `count_tokens(text: str) -> int` that returns `len(_get_encoder().encode(text))`
- `_truncate_text_to_tokens(text: str, max_tokens: int) -> str` helper that encodes, slices token list, decodes back. Append "..." if truncated.

**`truncate_list_by_token_budget()` function:**
- Signature: `(items: list[dict], text_key: str, max_tokens: int, max_item_tokens: int = 500) -> tuple[list[dict], int, int]`
- Returns: `(truncated_items, tokens_used, total_items_before_truncation)`
- Sort is NOT done here (caller sorts by relevance score before calling)
- Iterate items: count tokens of item[text_key], truncate individual item text if > max_item_tokens, accumulate total, break when budget exhausted
- Return the truncated list, tokens used, and original item count

**`deduplicate_chunks()` function:**
- Signature: `(chunks: list[dict], key: str = "segment_id") -> list[dict]`
- Remove duplicates by key, preserving order (first occurrence wins)
- If key is missing or empty string on an item, include it (don't skip items without IDs)

**`_calculate_chunk_budget()` function:**
- Accepts: max_total, entity_budget, relationship_budget, entity_tokens_used, relationship_tokens_used
- Returns: base_chunk_budget (max_total - entity_budget - relationship_budget) + unused from entities + unused from relationships
- Floor at 0 (never negative)

**`_build_context_string()` function:**
- Accepts: entities list[dict], relationships list[dict], chunks list[dict], graph_text str, total_entities int, total_relationships int, total_chunks int
- Returns the final Markdown string with:
  - Summary header: `Found N relevant entities, M relationships, and K document chunks` (only mention non-empty categories per Pitfall 4 in research)
  - `## Knowledge Graph Entities` section: bullet list `- **{name}**: {description}`, followed by `[+N more entities not shown]` if truncated
  - `## Knowledge Graph Relationships` section: VDB relationships formatted as `**{src_entity}** -> {rel_type} -> **{tgt_entity}**: {description}`, then graph_text appended as supplementary content (if non-empty), followed by `[+N more relationships not shown]` if VDB results were truncated
  - `## Document Chunks` section: each chunk with `[Source: {source_label}]` header and content
  - Omit empty categories entirely (no empty headers)
  - If ALL categories empty: return `"No relevant context found in knowledge base"`
- Truncation indicator counts: `total_raw - len(shown_items)` (uses pre-truncation count, not pre-dedup count)

**`assemble_context()` main function:**
- Signature: `(es_results: list, graph_text: str, entity_vdb_results: list[dict], rel_vdb_results: list[dict], budget: ContextBudget | None = None) -> AssembledContext`
- **Stage 1 (Collect):** Normalize ES results into dicts with keys: `segment_id`, `content`, `source_label` (built from document_title + section_path), `source_url`. Entity VDB results expected as dicts with `name`, `entity_type`, `description`, `score`. Relationship VDB results expected as dicts with `src_entity`, `tgt_entity`, `rel_type`, `description`, `score`.
- **Stage 2 (Truncate):** Sort each category by `score` descending (if present, fallback to insertion order). Call `truncate_list_by_token_budget` for entities (text_key="description"), relationships (text_key="description"), and chunks (text_key="content") with their respective budgets. For chunks, calculate dynamic budget via `_calculate_chunk_budget` using entity and relationship tokens_used.
- **Stage 3 (Dedup):** Call `deduplicate_chunks()` on the truncated chunk list.
- **Stage 4 (Build):** Call `_build_context_string()` with all truncated/deduped results.
- Log token budget usage at DEBUG level: `logger.debug("context_assembly_budget", entities="X/Y", relationships="X/Y", chunks="X/Y", total="X/Y")`
- Return `AssembledContext` with text and token counts.

**Important implementation notes:**
- Token counting is on RAW content text, NOT on formatted Markdown (per anti-pattern in research)
- ES SearchResult objects have attributes: `.content`, `.document_title`, `.section_path`, `.source_url`, `.segment_id`, `.source_id` -- normalize these in Stage 1
- Use `import structlog; logger = structlog.get_logger(__name__)` per project pattern
- graph_text (from Graphiti search) is pre-formatted text -- count its tokens toward relationship budget, include after VDB relationship items
