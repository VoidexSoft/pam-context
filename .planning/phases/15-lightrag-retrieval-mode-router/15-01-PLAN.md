<objective>
Create the two-tier query classifier module that categorizes user queries into 5 retrieval modes (entity, conceptual, temporal, factual, hybrid) using rule-based heuristics as the primary layer and an LLM fallback via Claude Haiku for ambiguous queries. Add config settings for all tunable parameters.

Purpose: Enables mode-based retrieval routing in smart_search (Plan 02), so that entity-specific questions skip relationship search, factual questions skip the graph entirely, and ambiguous queries fall back to searching everything. Classification happens in <1ms for 60-70% of queries (rule-based) with LLM fallback adding 500ms-2s only for the remaining uncertain queries.

Output: `src/pam/agent/query_classifier.py` module, config settings, and comprehensive unit tests.
</objective>

**2. Create `src/pam/agent/query_classifier.py`:**

This module exports: `classify_query_mode`, `RetrievalMode`, `ClassificationResult`.

**RetrievalMode enum (str, Enum):**
```python
class RetrievalMode(str, Enum):
    """Retrieval strategy modes for query routing."""
    ENTITY = "entity"           # Shallow graph: ES + entity VDB
    CONCEPTUAL = "conceptual"   # Deep graph: Graphiti + relationship VDB
    TEMPORAL = "temporal"       # All paths with temporal focus
    FACTUAL = "factual"         # ES-only, skip graph entirely
    HYBRID = "hybrid"           # All 4 retrieval paths (default)
```

**ClassificationResult dataclass:**
```python
@dataclass
class ClassificationResult:
    mode: RetrievalMode
    confidence: float  # 0.0 - 1.0
    method: str        # "rules", "llm", or "default"
```

**`classify_query_mode()` async function â€” the main entry point:**
- Signature: `async def classify_query_mode(query: str, client: AsyncAnthropic | None = None, vdb_store: EntityRelationshipVDBStore | None = None) -> ClassificationResult`
- Loads settings via `get_settings()` for threshold, keyword lists, and LLM toggle
- **Step 1:** Call `_rule_based_classify(query, settings)`. If result.confidence >= threshold, log and return.
- **Step 2:** If vdb_store is not None, call `await _check_entity_mentions(query, vdb_store)`. If result is not None and confidence >= threshold, log and return.
- **Step 3:** If client is not None AND `settings.mode_llm_fallback_enabled` is True, call `await _llm_classify(query, client)`. If result.confidence >= threshold, log and return.
- **Step 4:** Default: return `ClassificationResult(RetrievalMode.HYBRID, 0.5, "default")`.
- After determining the result (at any step), log via: `logger.info("query_mode_classified", mode=result.mode.value, confidence=result.confidence, method=result.method, query=query[:100])`

**`_rule_based_classify()` function (synchronous):**
- Signature: `def _rule_based_classify(query: str, settings: Settings) -> ClassificationResult`
- Parse keyword lists from settings by splitting on comma: `settings.mode_temporal_keywords.split(",")`
- Build regex patterns from parsed keywords (strip whitespace, wrap with `\b` word boundaries where appropriate)
- Classification priority order (highest specificity first):
  1. **Temporal:** Count temporal keyword regex matches in lowered query. 2+ matches -> confidence 0.9. 1 match -> confidence 0.75.
  2. **Factual:** Match factual patterns (anchored to start of query with `^`). Any match -> confidence 0.8. BUT apply negative signal: if the query also contains conceptual keywords or likely entity mentions (2+ consecutive capitalized words), reduce confidence to 0.5 to avoid the "What teams use Kubernetes?" pitfall.
  3. **Conceptual:** Count conceptual keyword matches. 2+ matches -> confidence 0.85. 1 match -> confidence 0.7.
  4. **No confident match:** Return `ClassificationResult(RetrievalMode.HYBRID, 0.4, "rules")`

**`_extract_candidate_names()` function (synchronous):**
- Signature: `def _extract_candidate_names(query: str) -> list[str]`
- Extract potential entity name candidates from the query:
  - Find sequences of 2+ consecutive capitalized words (e.g., "Auth Service", "Sales Team") using regex `r'\b([A-Z][a-z]+(?:\s+[A-Z][a-z]+)+)\b'`
  - Find single capitalized words that are NOT at the start of a sentence and are at least 3 chars (avoid "I", "A", etc.) â€” but only include these if they are NOT common English words. Use a small stop set: {"The", "This", "That", "What", "When", "Where", "How", "Who", "Which", "Does", "Did", "Can", "Could", "Would", "Should", "Has", "Have", "Had"}
  - Also find PascalCase compound words (e.g., "AuthService", "PaymentGateway") using regex `r'\b([A-Z][a-z]+[A-Z][a-zA-Z]*)\b'`
- Return a deduplicated list of candidate names

**`_check_entity_mentions()` async function:**
- Signature: `async def _check_entity_mentions(query: str, vdb_store: EntityRelationshipVDBStore) -> ClassificationResult | None`
- Call `_extract_candidate_names(query)`. If empty, return None.
- Query ES `pam_entities` index using a `terms` query on the `name` keyword field with the lowercased candidate names. Use `size: 1` since we only need to know if any match exists.
- **Important:** Access the ES client via `vdb_store.client` and the index name via `vdb_store.entity_index` (existing attributes on EntityRelationshipVDBStore).
- On match: return `ClassificationResult(RetrievalMode.ENTITY, 0.85, "rules")`
- On no match or any exception: return None (graceful degradation, entity check is best-effort)
- Wrap the entire ES call in try/except Exception to handle ES unavailability gracefully

**`_llm_classify()` async function:**
- Signature: `async def _llm_classify(query: str, client: AsyncAnthropic) -> ClassificationResult`
- Follow the same pattern as `extract_query_keywords()` in keyword_extractor.py:
  - Use model `"claude-3-5-haiku-20241022"` (hardcoded, same as keyword extractor)
  - max_tokens=50 (classification is ~20-30 tokens of JSON output)
  - timeout=10.0 (shorter than keyword extraction since output is simpler)
- Prompt template (system-style, inline with user message):
  ```
  Classify this query into exactly one retrieval mode.
  Modes:
  - entity: about a specific named entity, its properties or neighbors
  - conceptual: about relationships, dependencies, patterns between entities
  - temporal: about changes over time, history, before/after comparisons
  - factual: simple definition or fact lookup, answerable from documents alone
  - hybrid: unclear or spans multiple modes

  Output JSON only: {"mode": "<mode>", "confidence": <0.0-1.0>}

  Query: "{query}"
  ```
- Parse JSON response. Map mode string to RetrievalMode enum. Return with method="llm".
- On any error (JSON parse, timeout, API error): log warning and return `ClassificationResult(RetrievalMode.HYBRID, 0.5, "llm")` â€” graceful degradation to hybrid on LLM failure.

**Important implementation notes:**
- Import `from __future__ import annotations` at the top for modern type hints
- Use `from typing import TYPE_CHECKING` and put EntityRelationshipVDBStore import under `if TYPE_CHECKING:` to avoid circular imports
- Use `import structlog; logger = structlog.get_logger()` per project pattern
- The `name` field in the `pam_entities` ES index mapping is typed as `keyword` (exact match). The ES `terms` query requires lowercase values to match keyword fields that store as-is. Since entity names are stored as-is (e.g., "AuthService"), the terms query should use the original casing from `_extract_candidate_names()`, NOT lowercased.
